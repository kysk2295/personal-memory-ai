# AI 개인기억 에이전트 서비스 PRD

## 1. 최종 목적

이 서비스의 최종 목적은 사용자가 일상에서 남긴 일기, 고민, 감정, 프로젝트 기록, 대화, 의사결정 데이터를 기반으로 “나보다 나를 더 잘 이해하는 AI”를 만드는 것이다.

단순한 일기 앱이나 세컨브레인 툴이 아니라, 사용자의 기억과 사고 패턴을 장기적으로 학습한 개인 AI 에이전트가 고민을 함께 분석하고, 반복되는 문제를 발견하며, 더 나은 선택을 돕는 개인 기억 기반 AI 시스템을 지향한다.

이때 가장 중요한 원칙은 사용자의 일기와 세컨브레인이 기본적으로 개인에게만 접근 가능해야 한다는 점이다. 일기는 가장 민감한 개인 데이터이므로, 모든 기억을 클라우드에 무조건 저장하고 서버가 전부 소유하는 방식은 지양한다. 사용자의 기억은 사용자가 통제하고, AI는 허락된 범위 안에서만 기억을 읽고 활용해야 한다.

## 2. 서비스 한 줄 정의

앱에서 일기를 쓰면 관련된 과거 기억이 함께 연결되고, 웹에서 각자만 접근 가능한 개인 세컨브레인으로 기억 흐름을 탐색하며, AI 에이전트가 사용자의 장기 기억을 학습해 고민 해결을 돕는 프라이빗 개인 기억 서비스.

## 3. 제품 구조

서비스는 크게 3개의 축으로 구성된다.

```text
모바일 앱
→ 개인 일기와 생각 기록

웹 세컨브레인
→ 개인 전용 기억 그래프, 검색, 연결, 회고

AI 에이전트
→ 개인 기억 학습, 고민 분석, 해결 제안
```

## 4. 핵심 컨셉

### 4.1 앱: 개인 일기 입력 공간

앱은 사용자가 가장 쉽게 기록하는 공간이다. 사용자는 완성된 글을 쓰지 않아도 된다. 짧은 감정, 오늘의 고민, 대화 중 떠오른 생각, 프로젝트 관련 판단을 채팅하듯 입력한다.

앱의 핵심은 “기록 부담을 줄이는 것”이다.

예시:

```text
오늘 팀원이랑 의견이 안 맞아서 답답했다.
내가 또 혼자 해결하려고 한 것 같다.
```

AI는 이 기록을 다음처럼 구조화한다.

```text
감정: 답답함, 부담
주제: 팀 갈등, 책임감, 혼자 해결하려는 패턴
관련 기억: 3월 프로젝트 지연 회고, 5월 역할 분담 고민
회고 질문: 왜 도움을 요청하기보다 혼자 해결하려 했을까?
```

### 4.2 웹: 세컨브레인과 연관 기억 탐색 공간

웹은 기록된 기억을 더 넓게 다루는 개인 전용 세컨브레인 공간이다. 앱이 입력 중심이라면, 웹은 정리, 탐색, 연결, 회고 중심이다.

참고/벤치마크 방향은 Career Hacker Alex의 Second Brain 페이지처럼 노드와 엣지로 지식 그래프를 보여주는 방식이다.

- 벤치마크 URL: `https://www.careerhackeralex.com/memory`
- 벤치마크해야 할 핵심 UX: 다크 풀스크린 그래프 캔버스, 좌측 노드/엣지 유형 필터와 카운트, 상단 ask/search bar, 노드 클릭 시 상세 패널, 검색/질문 결과의 그래프 하이라이트, 레이아웃 전환.
- 단순 참고 수준이 아니라 웹 세컨브레인 UX의 기준 레퍼런스로 삼는다.
- 다만 본 서비스에서는 공개 지식 그래프가 아니라, 사용자 본인만 접근할 수 있는 비공개 일기/기억 그래프가 되어야 한다.

웹에서는 사용자가 자신의 기억을 다음 단위로 볼 수 있다.

- 감정별 기억
- 프로젝트별 기억
- 사람별 기억
- 고민별 기억
- 의사결정 타임라인
- 반복 패턴
- AI가 발견한 연결 관계

핵심은 일기를 쓸 때 현재 기록과 연결되는 과거 기억이 함께 떠오르는 것이다. 여기서 “공유”는 다른 사람에게 공개한다는 뜻이 아니라, 하나의 일기와 여러 과거 기억이 같은 맥락 안에서 함께 연결된다는 의미다.

예시:

```text
오늘의 일기:
팀원이랑 의견이 안 맞아서 답답했다.

연관된 과거 기억:
- 3월: 출시 직전 역할 분담이 흐려졌던 기록
- 4월: 혼자 해결하려다 번아웃이 왔던 기록
- 5월: 도움을 요청했을 때 오히려 일정이 빨라졌던 기록

AI 해석:
이번 고민은 단순한 팀 갈등보다 “역할 불명확 + 혼자 책임지려는 패턴”과 관련이 있습니다.
```

즉, 사용자가 새 일기를 쓰면 그 일기는 혼자 떨어진 기록이 아니라 과거 기억들과 함께 하나의 맥락 덩어리로 묶인다.

### 4.3 AI 에이전트: 나를 이해하는 고민 해결 AI

AI 에이전트는 단순히 질문에 답하는 챗봇이 아니다. 사용자의 과거 기록, 감정 패턴, 의사결정 이력, 인간관계 맥락, 프로젝트 경험을 참고해 현재 고민을 분석한다.

예시:

사용자 질문:

```text
이번에도 기능을 줄여야 할까, 아니면 더 만들어야 할까?
```

AI 응답 방향:

```text
과거 기록을 보면 출시 직전 기능을 늘렸을 때 일정이 밀린 경우가 2번 있었습니다.
반대로 4월에는 기능을 줄이고 핵심 플로우만 검증했을 때 실행 속도가 빨라졌습니다.

이번 상황은 3월 출시 지연 패턴과 더 비슷합니다.
따라서 새 기능 추가보다 핵심 기능 검증을 먼저 추천합니다.
```


## 7. 연관 기억 설계

연관 기억은 이 서비스의 가장 중요한 차별점이다. 사용자가 새 일기를 작성하면 AI가 과거 일기와 기록 중 의미적으로 연결되는 기억을 찾아 현재 기록 옆에 함께 보여준다.

### 연관 단위

- 비슷한 감정
- 비슷한 고민
- 같은 사람과의 관계
- 같은 프로젝트
- 반복된 의사결정 상황
- 과거의 실패 또는 성공 경험

### 연관 기억의 역할

- 사용자가 잊고 있던 과거 기록을 다시 불러온다.
- 현재 고민이 처음이 아니라 반복된 패턴인지 보여준다.
- 과거의 선택과 결과를 현재 의사결정에 활용한다.
- AI 에이전트가 사용자를 더 깊게 이해할 수 있는 근거가 된다.

### 화면 예시

```text
오늘의 기록
“또 프로젝트 마감 전에 기능을 더 넣고 싶어졌다.”

연관 기억
1. 2026.03.12 - 출시 직전 기능을 늘렸다가 일정이 밀림
2. 2026.04.08 - 기능을 줄였더니 테스트 속도가 빨라짐
3. 2026.05.01 - 불안할 때 범위를 늘리는 패턴을 회고함

AI 코멘트
이번 기록은 “마감 전 불안 → 기능 추가 욕구 → 일정 리스크” 패턴과 연결됩니다.
```

### 민감 정보 보호

기억은 기본적으로 개인에게만 보인다. 이 서비스의 핵심은 타인에게 공개하는 것이 아니라, 사용자의 현재 일기와 과거 기억을 안전하게 연결하는 것이다.

외부 공유 기능은 MVP에서 제외하며, 필요하더라도 이후 선택 기능으로만 검토한다.

## 8. AI 학습 방향

초기에는 실제 모델 파인튜닝보다 RAG 기반 개인 기억 시스템으로 시작하는 것이 적합하다.

### 1단계: RAG 기반 개인 기억

- 사용자의 기록을 임베딩으로 저장
- 질문이 들어오면 관련 기억 검색
- 검색된 기억을 바탕으로 AI가 답변

### 2단계: 사용자 피드백 학습

- 사용자가 AI 답변에 대해 “도움 됨”, “아님”, “기억 부정확” 피드백
- AI가 선호하는 말투, 조언 방식, 회고 스타일을 학습
- 사용자별 응답 스타일 개인화

### 3단계: 개인 에이전트화

- 사용자의 장기 목표 이해
- 반복되는 고민 선제 감지
- 중요한 결정 전 과거 패턴 알림
- 프로젝트별 조언 제공

### 4단계: 고도화된 개인 모델

충분한 데이터와 명확한 동의가 있을 때만 개인화 모델 학습 또는 미세조정을 검토한다.

단, 민감 정보가 많기 때문에 초기 제품은 파인튜닝보다 검색 기반 기억 시스템이 더 안전하다.

## 8-1. 프라이버시와 저장 원칙

이 서비스는 일기를 다루기 때문에 일반적인 생산성 SaaS보다 훨씬 강한 프라이버시 원칙이 필요하다.

### 핵심 원칙

- 기본값은 비공개다.
- 각 사용자의 세컨브레인은 각자만 접근할 수 있다.
- 서버는 사용자의 기억을 마음대로 학습하거나 열람하지 않는다.
- AI가 기억을 사용할 때는 사용자의 명시적 요청 또는 설정 범위 안에서만 접근한다.
- 외부 공유 기능은 핵심 기능이 아니며, MVP에서는 제외한다.

### 저장 방식 방향

초기 제품에서는 완전한 로컬 앱보다 개발 속도를 위해 서버 저장을 사용할 수 있지만, 장기적으로는 로컬 우선 또는 사용자 소유 저장소 구조를 지향한다.

권장 구조:

```text
사용자 기기
원본 일기, 민감 기억, 개인 키
↓
암호화된 동기화 저장소
백업과 기기 간 동기화
↓
AI 처리 계층
사용자가 허용한 기억만 일시적으로 검색/분석
↓
개인 세컨브레인
사용자 본인만 접근 가능한 기억 그래프
```

### 가능한 저장 옵션

1. 로컬 우선 저장
   - 원본 일기는 기기에 저장
   - 웹 접근을 위해 암호화된 동기화만 제공
   - 사용자가 데이터 삭제와 내보내기를 완전히 통제

2. 암호화 클라우드 저장
   - 서버에는 암호화된 데이터만 저장
   - 복호화 키는 사용자에게 귀속
   - 운영자가 원문을 볼 수 없는 구조 지향

3. 개인 저장소 연동
   - Obsidian vault
   - iCloud Drive
   - Google Drive
   - 개인 서버 또는 Supabase self-hosted

### AI 접근 방식

AI는 사용자의 전체 기억을 항상 읽는 존재가 아니라, 질문이나 일기 작성 상황에 맞춰 필요한 기억만 검색해 참고하는 구조여야 한다.

```text
새 일기 작성
↓
로컬 또는 암호화 저장소에서 관련 기억 검색
↓
최소한의 관련 기억만 AI 컨텍스트로 전달
↓
AI가 연관 기억과 회고 생성
↓
결과는 다시 사용자 개인 공간에 저장
```

## 8-2. 개인 세컨브레인 웹 구조

웹 세컨브레인은 각 사용자에게 개인 URL 또는 로그인 기반 공간으로 제공된다. 외부에서 접근할 수 있는 공개 페이지가 아니라, 본인 인증 후에만 열리는 개인 기억 대시보드다.

### 화면 방향

참고 사이트처럼 다음 요소를 제공한다.

## 13. 기술 아키텍처

```text
Mobile App
React Native / Flutter
↓
API Server
FastAPI / NestJS
↓
PostgreSQL
암호화된 원본 기록, 사용자, 기억 연결 저장
↓
Vector DB
Qdrant / Pinecone
↓
LLM Layer
기억 검색, 요약, 회고, 고민 분석
↓
Web App
Next.js 기반 세컨브레인 대시보드
```

장기적으로는 Vector DB에도 원문을 그대로 넣지 않고, 필요한 최소 정보와 임베딩만 저장하는 구조가 바람직하다. 원본 일기는 사용자 통제 영역에 두고, AI 검색에는 요약 또는 암호화된 파생 데이터를 활용한다.

## 19. 참고 레퍼런스

### Career Hacker Alex - Second Brain

참고 URL: https://www.careerhackeralex.com/memory

해당 페이지는 Second Brain을 지식 그래프 형태로 보여준다. 노드와 엣지 수를 표시하고, 노드 유형과 엣지 유형을 기준으로 필터링하며, 사용자가 노드를 드래그하고 확대/축소하면서 본문을 열람할 수 있는 구조다.

본 서비스는 이 구조를 참고하되, 공개 지식 그래프가 아니라 사용자 본인만 접근 가능한 개인 기억 그래프로 재해석한다.

차이점:

- 참고 사이트: 공개형 개인 지식 그래프
- 본 서비스: 로그인한 사용자 본인만 접근 가능한 비공개 기억 그래프
- 참고 사이트: 지식과 통찰 중심
- 본 서비스: 일기, 감정, 고민, 의사결정, 반복 패턴 중심
- 참고 사이트: 지식 노드 탐색
- 본 서비스: 현재 일기와 과거 기억의 자동 연결 및 AI 고민 해결

## 20. 결론

이 서비스는 일기 앱, 세컨브레인, AI 상담, 개인 에이전트가 결합된 형태다. 사용자는 앱에서 자연스럽게 일기를 쓰고, 웹에서 자신의 기억이 어떻게 연결되는지 탐색하며, AI 에이전트는 그 기억을 기반으로 사용자의 고민을 해결한다.

초기에는 “기록을 남기면 AI가 과거의 나를 불러와 현재의 고민을 도와준다”는 경험을 만드는 것이 가장 중요하다. 장기적으로는 사용자의 감정, 관계, 프로젝트, 목표, 의사결정 패턴을 이해하는 개인 AI 에이전트로 발전할 수 있다.
